## Summary

Solid, consistent UI code with good React Query usage and clear visual hierarchy, but has several real issues: duplicate queries between parent and children, no query error handling surfaced to the user, missing accessibility on interactive elements, loose string types throughout the API layer, and one correctness bug in the search query key.

## Issues

- **[severity: high]** Correctness: `SwarmCatalog` has a search bug. The `queryKey` includes `search` (`["swarm", "capabilities", selectedCompanyId, search]`) but the parent `Swarm.tsx` also fetches the same base key `["swarm", "capabilities", selectedCompanyId]` without `search`. When the user types in the search box, React Query fires a new fetch with the search term, but the parent cache entry (without the search param) is a different key — so `capCount` shown in the nav badge always reflects the unfiltered count, which is fine, but more importantly the client-side `filtered` array in `SwarmCatalog` re-filters data that was already filtered server-side. If the server honors the `search` param, the client `filter` pass runs on already-narrowed data, which means type/pricing filters applied after a search will operate on a subset rather than the full dataset. The search and client-side filtering should either both be client-side (fetch everything once, filter locally) or both be server-side (pass all active filters to the API). Mixing them silently produces wrong results when both are active.

- **[severity: high]** Performance: Triple duplicate fetches on mount. `Swarm.tsx` (the parent) independently fetches `listInstalls`, `listSources`, and `listCapabilities`. Each child component (`SwarmMySwarm`, `SwarmSources`, `SwarmCatalog`) also fetches the same data with the same query keys. React Query deduplicates in-flight requests, but only when keys are identical — `SwarmCatalog` adds `search` to its key so it diverges immediately. For the others, having the parent pre-fetch is reasonable as a priming strategy, but the intent is unclear and creates maintenance risk: if a child changes its key, the parent silently stops warming it. Remove the three queries from `Swarm.tsx` and let each child own its data, or extract all queries to a single context/hook that children read from.

- **[severity: medium]** Error handling: No query errors are shown to the user in any component. All five components destructure only `{ data, isLoading }` from `useQuery` and ignore `isError`/`error`. A failed fetch silently shows the empty-state UI ("No capabilities found", "No sources configured") which looks like a real empty state. At minimum, add `isError` handling that shows a distinct message: `if (isError) return <ErrorState />`. This applies to `SwarmCatalog`, `SwarmMySwarm`, `SwarmSources`, `SwarmAudit`, and the three queries in `Swarm.tsx`.

- **[severity: medium]** Type safety: Every domain-specific string field in the API types is typed as `string` rather than a union. `SwarmCapability.capabilityType`, `trustLevel`, `pricingTier`; `SwarmInstall.status`, `capabilityType`, `pricingTier`; `SwarmAuditEntry.action`, `actorType`; `SwarmSource.sourceType`, `trustLevel` — all are plain `string`. The color/style lookup maps (`TYPE_COLORS`, `TRUST_COLORS`, etc.) then need defensive `?? fallback` on every access, and typos in data are silent. Define string unions for each (`type CapabilityType = "skill" | "mcp" | "connector" | "plugin"`, etc.) and use them in both interfaces and lookup maps. This eliminates the fallback defensiveness and makes the lookup maps exhaustively typed.

- **[severity: medium]** Accessibility: The `CapabilityCard` in `SwarmCatalog.tsx` (line 35) is a `<div>` with `cursor-pointer` but no `role`, `tabIndex`, `onKeyDown`, or `aria-label`. It is not keyboard-reachable and is invisible to screen readers. Since it appears to be a clickable detail trigger, it should be a `<button>` (or an `<a>` if it navigates). Same issue applies to any future card that uses the same pattern.

- **[severity: medium]** Accessibility: Navigation buttons in `Swarm.tsx` (line 70) render emoji as the visible icon with no screen-reader alternative. The emoji (`\u{1F4E6}`, `\u{1F41D}`, etc.) are read literally by screen readers ("package", "honeybee") which does not match the label intent. Wrap each icon in `<span aria-hidden="true">` and rely on the visible `{item.label}` text as the accessible name. The button already has a label child, so adding `aria-hidden` to the icon span is sufficient.

- **[severity: medium]** Accessibility: The tables in `SwarmMySwarm` and `SwarmSources` have no `<caption>` or `aria-label`. A screen reader announces these as anonymous tables with no context. Add `aria-label="Installed capabilities"` / `aria-label="Swarm sources"` to each `<table>`.

- **[severity: low]** Accessibility: The status indicator dot (`<span className="w-1.5 h-1.5 rounded-full ..." />`) in `InstallRow` and `SwarmSources` conveys status purely through color with no text alternative for colorblind users. The status text label next to it does provide text, so this is low severity, but the color dot itself should have `aria-hidden="true"` to avoid the screen reader announcing an empty span.

- **[severity: low]** Readability: `SwarmMySwarm.tsx` line 79 splits a combined Tailwind class string at runtime to extract just the color class: `meta.color.split(" ")[0]`. `meta.color` is defined as `"text-purple-400 border-purple-500/30"` — this works today but is fragile; reordering the string or adding a class breaks it silently. Extract `textColor` and `borderColor` as separate fields in `TYPE_META`.

- **[severity: low]** Type safety: `swarm.ts` line 5 builds a query string with manual string concatenation (`\`${k}=${encodeURIComponent(...)}\``). This is correct but the `qs` helper is duplicated from other API files if they exist. More importantly, the function silently drops keys with `undefined` values but does not handle `null` — a `null` value would be serialized as the string `"null"`. Either document that callers must not pass `null`, or add a `v !== null` guard alongside `v !== undefined`.

## Verdict

NEEDS CHANGES — the search+filter mixing bug (high), duplicate fetch architecture (high), and silent error states (medium) should be fixed before this ships. The accessibility issues on the card and navigation icons are also medium priority given this is a board-operator UI.
