## Summary

The split is mostly correct and the $ pattern is wired consistently, but there are three real issues: one missing $ assignment that causes a runtime crash, one use of an undefined $ reference, and one nested try/catch structural problem in executeRun that creates an unreachable inner catch.

## Issues

- **[severity: high]** Correctness — `$.budgets` is assigned **after** `Object.assign($, runOps)`, but `runOps` (heartbeat-run-ops.ts) reads `$.budgets` inside `claimQueuedRun` (line 336: `$.budgets.getInvocationBlock(...)`) and also inside `updateRuntimeState` (line 572: `costService(db, $.budgetHooks)`). `claimQueuedRun` is safe because it is only ever called at runtime after the factory returns. However `$.budgetHooks` is read in `updateRuntimeState` (line 572 of heartbeat-run-ops.ts) and **`$.budgetHooks` is never assigned to `$` anywhere in the stub**. Only `$.budgets` is assigned (line 105 of heartbeat.ts). This will throw `Cannot read properties of undefined (reading 'onCostEvent')` (or similar) the first time any run completes. Fix: either assign `$.budgetHooks = { ... }` before the runOps module is initialised, or replace `$.budgetHooks` inside `updateRuntimeState` with `$.budgets` (whichever matches the actual interface of `costService`).

- **[severity: high]** Correctness — In `heartbeat-execution.ts` `executeRun`, the function has **two nested try blocks where the outer catch is structured as a fallback for the inner one, but the inner try-catch already catches all errors thrown inside it and calls `$.finalizeAgentStatus`**. When the inner catch block itself throws (e.g. a DB error during `finalizeAgentStatus`), control jumps to the outer catch. The outer catch then calls `$.finalizeAgentStatus` again and `$.releaseIssueExecutionAndPromote` on a run that may already have been finalized. This double-finalization is a logic change from a flat single try/catch in the original: the original factory had one catch that handled all failures. The outer try wraps lines 77–971 with an inner try at line 467–937. The intent (catching setup failures before adapter.execute) is correct but the implementation allows the inner catch's own failures to re-enter the outer catch, which then double-calls finalization helpers. Fix: wrap only the setup section (before the inner try) in the outer try, or move the outer catch to cover only the pre-adapter setup lines.

- **[severity: medium]** Correctness — `heartbeat-wakeup.ts` `releaseIssueExecutionAndPromote` calls `$.resolveSessionBeforeForWakeup` (line 122). This function is exported from `heartbeat-session.ts` and assigned to `$` via `Object.assign($, sessionOps)` on line 87 of the stub — that is fine. However, `releaseIssueExecutionAndPromote` also calls `$.startNextQueuedRunForAgent` (line 180). `startNextQueuedRunForAgent` is in `runOps` which is assigned to `$` on line 93 of the stub, and `wakeupOps` is created on line 98 — **after** runOps. At factory construction time there is no problem because `createWakeupOps` only captures the `$` reference, not its current keys. The call happens at runtime when all keys are present. This is safe. No fix needed, but the ordering dependency is worth being explicit about.

- **[severity: medium]** Correctness — `heartbeat-cancellation.ts` `cancelRunInternal` calls `$.appendRunEvent(cancelled, 1, ...)` (line 136). The hardcoded sequence number `1` is a regression: in the original factory, all callers of `appendRunEvent` used `await nextRunEventSeq(run.id)` to get the next seq. Using `1` will cause a unique-constraint violation on `heartbeatRunEvents(runId, seq)` for any run that already has a seq=1 event (which all runs do — `executeRun` appends "run started" at seq=1). Fix: change to `await $.nextRunEventSeq(cancelled.id)`.

## Verdict

NEEDS CHANGES — the missing `$.budgetHooks` reference and the hardcoded `seq=1` in `cancelRunInternal` are both runtime bugs that will surface in production.
